SECTION 1

Documentation Overview

A complete inventory of every section in the SCC-AC documentation — what it covers and why it matters for implementation.

What is SCC-AC?

SCC-AC (Salesforce Contact Center with Amazon Connect) is an AWS-built managed package and supporting AWS infrastructure that extends the standard Salesforce Service Cloud Voice (SCV) BYOA integration. While SCV handles voice calls natively, SCC-AC adds Amazon Connect Chat, unified routing for Cases, observability logging, post-contact AI summaries, and an AI agent assistant — all surfaced inside Salesforce's Omni-Channel and Lightning Experience.

In plain English: SCV makes phone calls work in Salesforce. SCC-AC makes everything else (chat, digital channels, cases, AI) also work via Amazon Connect, inside Salesforce.

SECTION 2

AWS Components & Concepts

Every AWS service used by SCC-AC — what it is, why it's used, what data flows through it, and the Salesforce equivalent.

Core AWS Services Used by SCC-AC

MUST KNOW Amazon Connect

AttributeDetail
What it isAWS's cloud contact center platform. A fully managed service for handling inbound/outbound voice calls and chat. Think of it as the "telephony engine" living in AWS.
Why SCC-AC uses itIt IS the contact center. Amazon Connect handles the actual call/chat routing, queuing, IVR flows, and real-time communications. Salesforce is just the agent interface on top.
When usedEvery single contact interaction — inbound chat, inbound voice, outbound, transfers, routing decisions.
Data flowing through itVoice streams, chat messages, contact attributes (key-value pairs), contact IDs, queue assignments, agent state changes, Contact Trace Records (post-call metadata).
SF EquivalentThink of it as the Salesforce Contact Center platform itself — the equivalent of what Salesforce's own telephony would be if it were a separate cloud product.
Key sub-componentsContact Flows (routing logic), Queues (holding areas), Routing Profiles (agent capacity), Security Profiles (permissions), CCP (agent UI), Contact Lens (AI analytics)

MUST KNOW AWS Lambda

AttributeDetail
What it isServerless compute — you deploy code (functions) that run on-demand without managing servers. Each function runs for milliseconds to minutes and is billed per invocation. Cold starts exist but can be mitigated with Provisioned Concurrency.
Why SCC-AC uses itLambda is the bridge between Amazon Connect and Salesforce. When a contact flow needs to create a Salesforce record, route work, or process a CTR — it calls a Lambda function. Lambda then calls Salesforce APIs.
When usedOn every inbound chat (create session + route), on every contact disconnect (end session), on CTR arrival (import data), during setup (certificate generation, secret management).
Data flowing through itContact IDs, channel identifiers, Salesforce session tokens, routing decisions, CTR JSON records, certificate files, API responses.
SF EquivalentExactly like Apex @future methods or Queueable Apex — code that runs asynchronously in response to events, calls external APIs, and processes data. The key difference: Lambda code runs in AWS, not in Salesforce.

MUST KNOW AWS IAM (Identity & Access Management)

AttributeDetail
What it isThe security control layer for all of AWS. IAM defines WHO can do WHAT to WHICH resources. It has Users (humans/apps with credentials), Roles (assumed by services like Lambda), and Policies (JSON documents saying what actions are allowed/denied).
Why SCC-AC uses itEvery SCC-AC component needs permissions. Lambda functions need roles to call Connect and Secrets Manager. Salesforce needs an IAM User to call CloudFormation. Agents need an IAM Role to authenticate into Connect via SAML SSO.
When usedConstantly — every API call in AWS requires IAM authorization. Setup uses the SCCAC-Setup-User. Runtime uses Lambda execution roles. Agent SSO uses the ProvisioningRole.
Data flowing through itPolicy documents (JSON), access keys, secret keys, session tokens, SAML assertions, ARNs (Amazon Resource Names).
SF EquivalentThink Profiles + Permission Sets + Named Credentials combined. IAM Policies = Permission Sets (JSON instead of checkboxes). IAM Roles = system users with delegated access. IAM Users = service accounts with API credentials.
Important users createdSCCAC-Setup-User (setup + provisioning), SCCAC-Runtime-User (optional, for minimal runtime ops), ProvisioningRole (assumed by Salesforce to create Connect instance)

MUST KNOW AWS CloudFormation

AttributeDetail
What it isAWS's Infrastructure-as-Code service. You define all your AWS resources (Lambda functions, S3 buckets, IAM roles, etc.) in a template (JSON/YAML), and CloudFormation creates them all in the right order, handling dependencies automatically.
Why SCC-AC uses itSCC-AC provisions 20+ AWS resources per contact center. CloudFormation automates this so you don't have to manually create Lambda functions, IAM roles, Kinesis streams, etc. The Guided Setup wizard triggers CloudFormation deployments behind the scenes.
When usedDuring initial setup (6+ stack deployments), during upgrades (stack updates), during contact center enablement (per-CC stacks).
Data flowing through itTemplate definitions, parameter values (Org ID, CC ID, etc.), resource configurations, IAM policies, Lambda code references.
SF EquivalentLike the Salesforce Metadata API + Change Sets — it deploys a set of related components together, in the right order, with dependency management. But on the AWS side.

MUST KNOW AWS Secrets Manager

AttributeDetail
What it isA secure vault for storing secrets (passwords, API keys, certificates) that applications can retrieve at runtime. Supports automatic rotation and encryption via KMS.
Why SCC-AC uses itLambda functions need to call Salesforce APIs. They need the Connected App consumer key/secret and a Salesforce access token. These are stored in Secrets Manager so Lambda can fetch them securely at runtime without hardcoding credentials.
When usedOn every Lambda invocation that calls Salesforce — the Lambda fetches the cached access token from Secrets Manager, uses it, and refreshes it when expired.
Data storedConnected App consumer key + secret, certificate public/private keys, JKS password, SF API user username, cached Salesforce OAuth access tokens.
SF EquivalentExactly like Named Credentials + the credential vault part of External Credentials. Instead of Salesforce storing OAuth tokens, AWS Secrets Manager stores them for Lambda to use.
Key secretsSCCAC-Secret-{OrgId} (org-level: certs + Connected App creds) and SCCAC-InteractionApiAccessTokenSecret-{CCId} (per-CC: cached SF OAuth token)

Should Know — Supporting Services

SHOULD KNOW Amazon S3 (Simple Storage Service)

AttributeDetail
What it isObject storage — like a giant, infinitely scalable file system in the cloud. You store files (objects) in named containers (buckets). Accessed via HTTPS with IAM-controlled permissions.
Why SCC-AC uses itThree distinct purposes: (1) Store the generated JKS certificate file during setup for manual download. (2) Store chat attachment files when enabled. (3) Store observability logs (CCP init + agent activity) in a hierarchical structure.
SF EquivalentLike Salesforce Files / ContentDocument — a place to store binary files accessible via URL. The key difference: S3 uses IAM for access control rather than Salesforce sharing rules.
Key bucketsccac-{AWS_ACCOUNT_ID}-bucket-{SF_ORG_ID} — stores certificate JKS files. CORS must be configured on the attachments bucket for chat file uploads to work from Salesforce.

SHOULD KNOW Amazon CloudWatch

AttributeDetail
What it isAWS's observability service — collects logs, metrics, and events from all AWS services. You can create dashboards, set alarms, and query logs. Think of it as AWS's equivalent of Splunk/Datadog.
Why SCC-AC uses itSCC-AC automatically provisions a CloudWatch dashboard showing Lambda health metrics, and creates 10 alarms monitoring Lambda errors and throttles for each contact center.
When usedContinuously during operation. Lambda function logs go to CloudWatch log groups automatically. Alarms fire when error/throttle thresholds are exceeded.
SF EquivalentLike Salesforce Event Monitoring + Debug Logs + Health Check combined. CloudWatch = where you go to debug Lambda failures, just like Developer Console → Logs in Salesforce.
Key alarms10 alarms pre-created: error alarms (≥3 errors/min) and throttle alarms (≥5 throttles/min) for: CreateMessagingSession, Routing, CtrKinesisSegregation, ContactTraceRecord, ContactDisconnected functions.

SHOULD KNOW Amazon Kinesis Data Streams

AttributeDetail
What it isA real-time data streaming service. Think of it as a high-throughput message queue where data producers push records and consumers read them. Records are retained for 24h–7 days.
Why SCC-AC uses itAmazon Connect exports Contact Trace Records (CTRs — the post-contact metadata like duration, timestamps, agent info) via a Kinesis stream. The CtrKinesisRecordSegregationFunction Lambda reads these and imports them into Salesforce.
When usedAfter every contact ends — Amazon Connect emits a CTR to Kinesis, Lambda picks it up and imports it to Salesforce.
SF EquivalentLike Platform Events / Change Data Capture — an event stream where producers publish records and consumers read them asynchronously. CTRs via Kinesis = the AWS equivalent of Salesforce's CDC stream.

SHOULD KNOW Amazon EventBridge

AttributeDetail
What it isA serverless event bus. AWS services publish events (e.g., "contact disconnected in Amazon Connect"), and you create rules to route matching events to targets (e.g., a Lambda function).
Why SCC-AC uses itAmazon Connect publishes contact disconnect events to EventBridge. SCC-AC creates a rule that routes these events to the EndMessagingSessionFunction Lambda — providing an immediate signal when a contact ends, before the CTR is even emitted.
When usedImmediately when a contact disconnects (terminal disconnect: customer hang-up, system error, API stop). This is faster than waiting for the Kinesis CTR.
SF EquivalentLike a Salesforce Trigger or Platform Event subscription — an event fires, and code runs automatically. EventBridge = the subscription mechanism that says "when THIS event happens, run THIS function."

SHOULD KNOW AWS KMS (Key Management Service)

AttributeDetail
What it isManaged encryption key service. KMS generates and manages cryptographic keys used to encrypt data at rest. Other AWS services (like Secrets Manager) integrate with KMS to encrypt their stored data.
Why SCC-AC uses itSecrets Manager uses KMS to encrypt the secrets stored (Connected App credentials, cert keys, access tokens). SCC-AC creates its own KMS keys (prefixed SCC*) for this purpose.
SF EquivalentLike Salesforce Shield Platform Encryption — transparent encryption of stored sensitive data. You don't interact with KMS directly most of the time; it works under the hood.

SECTION 4

Identity & Access Deep Dive

Every user, service account, connected app, certificate, and permission set — what it is, why it exists, and exactly what conversation it belongs to.

The One Frame That Makes Everything Click

The confusion comes from trying to understand all these pieces in isolation. The key is to understand that there are exactly three separate conversations happening in this system. Every single user, app, and certificate exists to serve one of these three conversations — and only one.

#ConversationWho talks to whoWhen it happensWhat carries the identity
🔶 A — Provisioning Salesforce calls AWS to build infrastructure Once during initial setup; again when upgrading or adding a contact center SCCAC-Setup-User IAM access key + secret
🔷 B — Runtime AWS Lambda calls Salesforce to create/update records On every single contact — inbound chat, CTR import, session close awsscc GLOBAL_CONNECTED_APP + certificate
🟢 C — Agent Login Agent browser authenticates into Amazon Connect Every time an agent opens the Lightning App and becomes available {ContactCenter} Connected App SAML token
Read everything below through this lens. Every time you encounter a user, app, or certificate, ask yourself: which conversation does this belong to? A, B, or C? Once you can answer that, its purpose becomes obvious.

Users

There are two types of users in this system: human users (real people who log in) and machine users (service accounts that code uses to authenticate). They are completely separate and never overlap.

Connected Apps

There are three Connected Apps in this system. Two are created automatically by Salesforce. One you create manually. They serve three completely different purposes and are never interchangeable.

AUTO-CREATED Connected App 1: {ContactCenterName} Connected App

Created by: Salesforce automatically when you create a Contact Center in Partner Telephony Setup
Conversation: 🟢 C — Agent Login (SAML SSO)
Type: SAML Connected App
You configure: Manage Profiles — add System Administrator and Standard User

What it does: Lets agents log into Amazon Connect's CCP automatically — without seeing a separate login screen. When an agent opens the Lightning App, Salesforce uses this Connected App to generate a SAML assertion (a cryptographically signed ticket) and sends it silently to Amazon Connect. Connect validates it and logs the agent in. This is the only reason this Connected App exists.

AUTO-CREATED Connected App 2: {ContactCenterName} Connected App for RestAPI OAuth

Created by: Salesforce automatically alongside Connected App 1
Conversation: Internal SCV REST operations
You configure: Nothing
Docs say: "Ignore that one"

What it does: Handles internal REST API operations between Salesforce and the Amazon Connect instance (outside of SCC-AC's scope). You don't touch it. It is mentioned here only so you don't confuse it with Connected App 1 — they appear right next to each other in App Manager.

YOU CREATE Connected App 3: awsscc GLOBAL_CONNECTED_APP

Created by: You — by running Apex anonymous code in the install step
Conversation: 🔷 B — Runtime (Lambda → Salesforce)
Auth flow: OAuth 2.0 Client Credentials (Certificate-based)
Run As user: System Administrator by default; can be a dedicated minimum-access user

What it does: This is the OAuth gateway for AWS Lambda functions calling Salesforce. Every time Lambda needs to do something in Salesforce — create a MessagingSession, import a CTR record, end a session — it first presents this Connected App's consumer key and certificate to Salesforce's OAuth endpoint to get a short-lived access token. Salesforce issues the token. Lambda uses it for the API call.

The three values you must handle from this app:

Certificates

There are three certificates. They appear at different points in the setup and serve completely different purposes. The confusion arises because two of them are in Salesforce Certificate and Key Management at the same time.

YOU CREATE Certificate 1: Identity Provider Certificate

AttributeDetail
Created byYou, in Salesforce Setup → Identity → Identity Provider → Enable Identity Provider. You must create or select any certificate here.
PurposeEnables Salesforce to act as a SAML Identity Provider. Salesforce uses this certificate to digitally sign the SAML assertions it sends to Amazon Connect for agent SSO.
Conversation🟢 C — Agent Login only
Relationship to SCC-ACNothing to do with Lambda, AWS infrastructure, or the Connected App. It exists purely to make agent SSO work. The name does not matter — any self-signed certificate works.
After setupStays in Certificate and Key Management, never changes.

AUTO-CREATED Certificate 2: SCC_PLACE_HOLDER_CERTIFICATE

AttributeDetail
Created byThe Apex anonymous code — the createCertificate() function creates this automatically when you run the code in Developer Console.
PurposeA temporary placeholder. The awsscc GLOBAL_CONNECTED_APP Connected App requires a certificate to exist at the moment it is created. But the real certificate (from AWS Lambda) does not exist yet. So this placeholder is used to satisfy that requirement. It is never used at runtime.
ConversationNone at runtime — it is a setup artefact only
What happens to itGuided Setup Step 6 triggers Lambda functions that call Salesforce and replace this placeholder with the real certificate in both the Connected App and the External Credential. After Step 6, this certificate is redundant. You can leave it — it does nothing.

AttributeDetail

PurposeThe actual credential that proves Lambda's identity to Salesforce at runtime. Lambda holds the private key (in Secrets Manager). Salesforce holds the public key (in Certificate and Key Management). Together they form the Client Credentials authentication for the awsscc GLOBAL_CONNECTED_APP.
Conversation🔷 B — Runtime (Lambda → Salesforce), every single contact

Voice & Chat Journeys

Step-by-step flows showing exactly which AWS and Salesforce components participate at each stage.

Journey 1 — Inbound Chat (The Core SCC-AC Flow)

This is the primary SCC-AC use case. A customer initiates a chat and it flows through Amazon Connect → Lambda → Salesforce Omni-Channel.

1
[Customer] Opens chat widget (web/mobile). Client calls StartChatContact Amazon Connect API. Optional: passes clientId attribute for customer identity linking.
AWS components: Amazon Connect Chat API
2
[Amazon Connect] Contact arrives and is processed by the SCCAC - Sample Inbound Flow contact flow. The flow begins executing step by step.
AWS components: Amazon Connect Contact Flow engine
3
[Contact Flow Block: Set Contact Attributes] Sets channelAddressIdentifier (the SF Messaging Channel ID) and optionally EndMessagingSessionOnContactTermination=true.
Data: channelAddressId string, contact attributes map
4
[Contact Flow Block: Invoke Lambda] Calls SCCAC-CreateMessagingSessionFunction. Lambda reads channelAddressIdentifier from contact attributes, retrieves SF OAuth token from Secrets Manager, calls Salesforce Interaction Service API.
AWS components: Lambda, Secrets Manager. Data: contactId, channelAddressId → SF API call → returns three contact attributes: conversationIdentifier, workItemId, source=SCCAC
5
[Salesforce] A new MessagingSession record is created representing this chat contact. It appears as a work item pending routing in Omni-Channel.
SF components: Interaction Service API, MessagingSession object, Digital Engagement
6
[Contact Flow Block: Invoke Lambda] Calls SCCAC-RoutingFunction with routingType=Initial. Lambda calls SF API to trigger Omni-Channel routing on the MessagingSession. Returns externalQueueId = ARN of the mapped Amazon Connect queue.
AWS components: Lambda. SF components: Omni-Channel routing engine
7
[Contact Flow Block: Set Working Queue] Sets the Amazon Connect queue dynamically using the externalQueueId returned by the routing Lambda.
AWS components: Amazon Connect queues (e.g. SCC-BasicQueue)
8
[Contact Flow Block: Transfer to Queue] Contact enters the queue. Amazon Connect looks for an available agent with the right Routing Profile who covers this queue.
AWS components: Amazon Connect queue engine, routing profiles
9
[Salesforce] Omni-Channel detects agent is available (agent has Active status with Messaging channel). Presents the MessagingSession work item to the agent.
SF components: Omni-Channel, agent's Lightning App
10
[Agent] Accepts the work item in Omni-Channel. MessagingSession status changes to Active. The Enhanced Conversation widget in the Lightning page shows the chat transcript.
SF components: Omni-Channel, MessagingSession record, Enhanced Conversation widget
11
[Live Chat] Customer and agent exchange messages. Messages flow via Amazon Connect Chat API ↔ Salesforce Digital Engagement ↔ MessagingSession in real time.
AWS components: Amazon Connect Chat streaming. SF components: Digital Engagement
12
[Contact Ends] Agent or customer ends the chat. Two parallel cleanup processes begin immediately.
Triggers: EventBridge contact disconnect event + Kinesis CTR emission
13
[EventBridge → Lambda] Amazon Connect fires contact disconnect event → EventBridge routes to SCCAC-EndMessagingSessionFunction. This is the safety net — ensures MessagingSession is Ended in SF even if client-side close failed.
AWS components: EventBridge, Lambda. SF components: MessagingSession API
14
[Kinesis → Lambda → Salesforce] Amazon Connect emits Contact Trace Record (CTR) to Kinesis stream. CtrKinesisRecordSegregationFunction reads it, filters SCCAC contacts (source=SCCAC), invokes ContactTraceRecordFunction which imports CTR data into Salesforce.
AWS components: Kinesis, 2 Lambda functions. SF components: CTR object
15
[Optional: Post-Contact Summary] If Contact Lens enabled, an AI summary is generated. SF Apex Queueable job calls SCCAC-ContactTraceRecordFunction via ConnectLens API Named Credential, stores summary in MessagingSession PostContactSummary__c field.
AWS components: Contact Lens API. SF components: Queueable Apex, MessagingSession

Journey 2 — Inbound Voice Call (SCV + SCC-AC)

Voice is primarily handled by Service Cloud Voice (SCV), not SCC-AC. SCC-AC adds post-contact summaries and AI assistance on top.

1
[Customer] Dials a phone number claimed in Amazon Connect and mapped to a SCV inbound contact flow.
2
[Amazon Connect] SCV Inbound Contact Flow executes. Flow may include IVR prompts, bot interactions, Contact Lens analytics block, and Amazon Q AI block. Routes to a queue.
3
[Omni-Channel + SSO] Agent is available in Omni-Channel with Voice channel enabled. Salesforce SSO has already logged agent into CCP via SAML. Amazon Connect routes call to agent.
4
[Agent] CCP (hidden iframe in SF) rings → agent accepts in Omni-Channel. SCV automatically creates a VoiceCall record in Salesforce. Screen pop opens automatically.
5
[Live Call] Voice streams directly between customer ↔ Amazon Connect ↔ agent's CCP. If Amazon Q in Connect is configured, AI recommendations appear in the Assistant utility bar in real time.
6
[Call Ends] Agent wraps up. VoiceCall record updated by SCV. If Contact Lens + SCC-AC post-contact summary enabled: SF Apex Queueable calls Connect ContactLens API via Named Credential, stores AI summary in VoiceCall record.
Key distinction: For voice, SCV handles everything from call routing to VoiceCall record creation. SCC-AC only adds the AI summary feature on top of voice. For chat, SCC-AC handles everything (session creation, routing, CTR import).

Journey 3 — Case Routing (Unified Routing)

This is the "extend Salesforce Omni-Channel to Amazon Connect" feature — Cases can be routed to Amazon Connect agents, not just chat/voice contacts.

1
[Salesforce] A Case is created/updated and assigned to an Omni-Channel Queue that is: (a) configured with External Routing model and (b) mapped to an Amazon Connect queue in Partner Telephony Contact Centers.
2
[SCC-AC Package] Detects the queue assignment via a trigger/event. Calls the Connect API Named Credential to invoke connect:StartTaskContact in Amazon Connect — creating a "Task" type contact.
3
[Amazon Connect] Task contact arrives and is processed by SCCAC - Sample Omni-Channel Subflow - Task Routing. Task carries contact attributes: workItemId (Case ID), queueId (mapped Connect queue ARN), externalRouting=true.
4
[Contact Flow] Sets working queue from queueId attribute (or manually configured fallback) → Transfers to queue. Amazon Connect routes the Task to an available agent.
5
[Salesforce] Amazon Connect delivers work to agent → Omni-Channel presents the Case to the agent for acceptance. Agent sees the Case record, not a generic task.
6
[Agent] Accepts Case in Omni-Channel. Works the Case normally in Salesforce. When done, closes/updates Case as usual.

Journey 4 — Agent Login / Status Sync (SSO Flow)

1
[Agent] Logs into Salesforce normally.
2
[Agent] Opens the Lightning App with SCC-AC. The Amazon Connect CCP utility item starts automatically in a hidden iframe.
3
[Salesforce SAML SSO] The CCP loads and requests a SAML login from Salesforce. Salesforce (as SAML Identity Provider) generates a SAML assertion for this agent and sends it to Amazon Connect's SAML endpoint.
4
[Amazon Connect] Validates the SAML assertion, identifies the agent, and logs them into the CCP. The CCP iframe is now authenticated.
5
[Agent] Opens Omni-Channel in the utility bar, selects "Active" presence status.
6
[SCC-AC Package] Detects Omni-Channel status change → syncs status to Amazon Connect CCP via the hidden iframe. Agent now shows as Available in both systems simultaneously.
7
[Result] "Amazon Connect for Salesforce - Available" appears in the utility bar, confirming successful sync.
SECTION 7

AWS Learning Guide for Salesforce Architects

Every AWS concept ranked by importance, explained through Salesforce analogies. No AWS experience assumed.

Your mental model: Everything you know about Salesforce has an AWS equivalent. Apex = Lambda. Metadata API/Change Sets = CloudFormation. Named Credentials = Secrets Manager + IAM. Platform Events/CDC = Kinesis + EventBridge. Permission Sets = IAM Policies. Debug Logs = CloudWatch Logs.

MUST KNOW — Learn Before Implementation

1. IAM — Identity & Access Management
The AWS Permission System
SF Analogy: Profiles + Permission Sets + Named Credential authentication, but as JSON policies instead of checkbox UIs.

Key concepts: Users (humans/apps with long-lived creds), Roles (assumed by services — Lambda assumes a role), Policies (JSON: Effect/Action/Resource), ARNs (unique resource names like salesforce.com org IDs).

Why critical: Every SCC-AC failure that isn't a network issue is an IAM permissions issue. You will spend 30% of setup time on IAM. Know what an access key/secret key pair is, what a policy JSON looks like, and what "AssumeRole" means.
2. AWS Lambda
Serverless Functions = Apex in the Cloud
SF Analogy: Exactly like Apex @future or Queueable Apex — code that runs asynchronously in response to events, with no server to manage.

Key concepts: Functions are deployed as code packages. They have execution roles (IAM). They have timeout limits. Cold starts (first invocation is slower). Provisioned Concurrency pre-warms functions. Logs go to CloudWatch automatically.

Why critical: 10 Lambda functions ARE SCC-AC's backend. When chat routing fails, you look at Lambda logs in CloudWatch.
3. Amazon Connect
The Contact Center Engine
SF Analogy: Think of it as the underlying telephony platform — like if Salesforce Contact Center were a separate cloud product that you configure independently.

Key concepts: Instances (one per org), Contact Flows (the IVR/routing logic — like Flow Builder but for contacts), Queues (holding areas), Routing Profiles (per-agent capacity), Security Profiles (permissions in Connect), CCP (the agent UI), Quick Connects (pre-set transfer destinations).

Why critical: All chat and voice routing happens in Connect. You will regularly visit the Connect console to edit flows, check agent states, and debug routing issues.
4. CloudFormation
Infrastructure as Code = Metadata API
SF Analogy: Like Metadata API + Change Sets. You define what you want (in a template) and CloudFormation creates/updates/deletes it. Stacks = Change Sets. Template = Metadata XML. Stack Events = Deployment Status.

Key concepts: Stacks (a group of resources deployed together), Templates (JSON/YAML definitions), Stack Events (deployment log — check here when setup fails), Rollback (if any resource fails, the whole stack rolls back).

Why critical: When Guided Setup fails, you look at the CloudFormation stack's Events tab. Learn to read stack events to identify which resource failed and why.
5. Secrets Manager
Credential Vault = Named Credentials
SF Analogy: Like Named Credentials, but on the AWS side. Lambda functions fetch credentials from here at runtime rather than having them hardcoded.

Key concepts: Secrets are key-value JSON documents. Retrieve Secret Value (the one manual step after CloudFormation). Secret ARNs for cross-service references. Rotation (AWS recommends every 90 days — this is in the Appendix).

Why critical: The most common setup failure is forgetting to update the Secrets Manager secret with real values (replacing placeholders with the consumer secret and cert password).

SHOULD KNOW — Learn During Implementation

6. Amazon S3
Object Storage = Salesforce Files
SF Analogy: Like ContentDocument/ContentVersion — files stored and accessible via URL. But S3 is accessed via HTTP and controlled via IAM policies (not sharing rules).

Must know for SCC-AC: How to navigate to a bucket, find a file, and download it (needed for certificate download). How to configure CORS rules (needed for chat attachments). Bucket naming conventions are important — SCC-AC looks for buckets by prefix.
7. CloudWatch
Monitoring + Logs = Event Monitoring + Debug Logs
SF Analogy: Event Monitoring + Debug Logs + Health Check rolled into one. Lambda errors go here. Contact flow logs go here. CloudWatch alarms = workflow rules that email you when thresholds are breached.

Must know for SCC-AC: How to navigate to a Log Group (e.g., /aws/lambda/SCCAC-CreateMessagingSessionFunction-{CCId}), search log entries, and read Lambda execution logs. This is your primary debugging tool.
8. Amazon Kinesis
Real-time Streaming = Platform Events / CDC
SF Analogy: Like Platform Events or Change Data Capture — a stream where producers publish records and consumers read them asynchronously.

For SCC-AC: Amazon Connect publishes CTRs (Contact Trace Records) to a Kinesis stream. Lambda consumes these. You don't configure Kinesis directly — it's set up by CloudFormation. But knowing it exists explains why CTR data has a slight delay (usually seconds) before appearing in Salesforce.
9. Amazon EventBridge
Event Bus = Platform Event Subscriptions
SF Analogy: Like a Platform Event trigger — "when THIS event fires, call THIS function." EventBridge rules are the subscription mechanism.

For SCC-AC: A rule routes Amazon Connect contact disconnect events to the EndMessagingSessionFunction Lambda. This provides near-instant cleanup when contacts end. Created automatically by CloudFormation — you don't configure it, but knowing it exists explains why orphaned sessions get cleaned up quickly.
10. AWS KMS
Encryption Keys = Shield Encryption
SF Analogy: Like Salesforce Shield Platform Encryption — transparent encryption of data at rest. You rarely interact with KMS directly; it works under the hood when Secrets Manager encrypts your stored credentials.

For SCC-AC: SCC-AC creates its own KMS keys (in the SCC* namespace). The SCCAC-Setup-User IAM policy grants permissions to create and manage these keys. You'll see KMS referenced in the IAM policy JSON during setup.

NICE TO KNOW — Learn As Needed

11. SAML Federation
SSO Protocol for Agent Login
Know that Salesforce acts as the Identity Provider (IdP) for Amazon Connect. Agent SSO works via SAML 2.0 assertions. The most common SSO issue: multi-IdP mode and multi-session support in Connect must be DISABLED for SCV automated SSO to work.
12. AWS Step Functions
State Machines = Salesforce Flow
Referenced in the IAM setup policy with stateMachine permissions. May be used in the SCV provisioning workflow (outside SCC-AC's direct Lambda code). Like Salesforce Flow — defines a sequence of steps. You won't configure these directly.
13. Amazon Q in Connect
AI Agent Assistant = Einstein for Service
Real-time AI recommendations during contacts. Configured separately in the Amazon Connect console (Connect → Agent Assist). SCC-AC adds a Salesforce utility bar component that surfaces Q recommendations inside the Lightning App during live contacts.
14. Contact Lens
AI Analytics = Einstein Analytics for Calls
Enabled via the "Set recording and analytics behavior" block in contact flows. Provides transcription, sentiment analysis, and AI-generated post-contact summaries. The summaries flow via the ContactLens API Named Credential into MessagingSession / VoiceCall records in Salesforce.
15. Amazon SQS
Message Queue = Async Apex Queue
Simple Queue Service — buffers messages between producers and consumers. Used as a Lambda event source mapping in the CTR processing pipeline. Like the Salesforce Async Apex queue — prevents Lambda from being overwhelmed by CTR events during high volume. Created automatically by CloudFormation.

Final Summary

SCC-AC in simple English, the top pitfalls that kill implementations, and a practical learning roadmap.

SCC-AC Explained in Simple English

The 2-minute explanation:

Salesforce Service Cloud Voice lets agents handle phone calls inside Salesforce. But it doesn't natively support chats from Amazon Connect, and it doesn't let you route Cases through Amazon Connect's agent capacity.

SCC-AC is a managed package + AWS infrastructure that fills these gaps. When a customer starts a chat, an Amazon Connect contact flow calls a Lambda function which creates a MessagingSession record in Salesforce. Salesforce Omni-Channel then routes that session to the right agent. The agent handles the chat inside Salesforce. When the chat ends, Lambda imports the contact record data back into Salesforce.

Think of Lambda as the translator that speaks both "Amazon Connect" and "Salesforce" fluently. Everything else (S3, Secrets Manager, CloudWatch, EventBridge, Kinesis) is just the supporting infrastructure that makes Lambda work reliably at scale.

Common Implementation Pitfalls

PitfallSymptomRoot Cause & Fix
🔴 Not updating Secrets Manager placeholdersCertificate step fails; Lambdas can't authenticate to SalesforceAfter CloudFormation deploys global stack, manually edit SCCAC-Secret in Secrets Manager — replace both placeholder values with real consumer secret + cert password
🔴 Named Credentials not enabled for calloutsCallout failures; setup steps show no green checkmarkEvery Named Credential must have "Enable for Callouts" toggled ON manually. Check all 6+ Named Credentials after each Guided Setup step
🔴 Using sample contact flows directly without cloningConfiguration lost after SCC-AC package upgradeAlways Save As → clone → edit the copy. Sample flows get reset on upgrades. This is called out in the docs but easy to miss.
🔴 Forgetting to configure channelAddressIdentifier in the flowLambda errors; MessagingSession not created; chat goes nowhereAfter creating the Messaging Channel in SF, note the Channel Address Identifier. Update the Set Contact Attributes block in your cloned inbound flow with this value. Save AND Publish the flow.
🔴 Sandbox refresh breaks everythingSCC-AC stops working entirely after sandbox refreshSandbox refresh changes the Org ID, My Domain, and SCV settings. SCC-AC must be completely uninstalled and reinstalled after a sandbox refresh. There is no patch — full reinstall only.
🟡 Permission sets not updated after package upgradeAgents can't see new fields/objects added in the upgradeAfter each upgrade, compare SCC Administrator / SCC Agent with the SCC Administrator Placeholder / SCC Agent Placeholder. Manually add any new permissions to your cloned sets.
🟡 Multi-IdP or multi-session in Amazon Connect enabledAgents can't log into SCV — "We can't log you in" errorDisable multi-IdP and multi-session support in the Amazon Connect instance settings. These features require user interaction that breaks Salesforce's automated SSO flow.
🟡 Queue mapping not configuredChat contacts route to SCC-BasicQueue regardless of Omni-Channel setupIn SF Partner Telephony Contact Centers → Queue Mapping, map each SF Omni-Queue to its corresponding Amazon Connect queue. Without this, Omni-Flow routing returns no queue and falls back to BasicQueue.
🟡 Routing configuration model not set to External RoutingOne messaging session uses 100% of agent capacity in Omni-SupervisorChange the routing configuration used by the messaging queue to use the "External Routing" model. This lets Connect manage capacity instead of Salesforce.
🟡 Chat transfer missing Quick ConnectsTransfer button doesn't show destinationsCreate Quick Connects in Amazon Connect for each agent/queue you want to transfer to, AND add those Quick Connects to the queue's settings. Both steps are required.
🟡 IAM credentials not rotated at 90 daysSetup operations fail; potential production outageRotate SCCAC-Setup-User credentials every 90 days and update both the SF External Credentials and the AWS IAM user simultaneously. Document this as a recurring task.
🔵 SCCAC ApiAccessUser deactivated in SFLambda functions can't authenticate; all chat operations failThe SCCAC ApiAccessUser is created during Guided Setup Step 2 and is used by Lambdas to access Salesforce APIs. Never deactivate it. Monitor it proactively.

Practical Learning Roadmap

A week-by-week guide to becoming productive with Amazon Connect and SCC-AC as a Salesforce architect.

Week 1
AWS Fundamentals: IAM + Console Navigation Create a free AWS account. Complete AWS's free "IAM Essentials" course. Practice: create an IAM user, attach a policy, create an IAM role. Navigate to CloudFormation, Lambda, S3, Secrets Manager, CloudWatch. Read all 4 IAM policy JSONs in the SCC-AC prerequisites section and understand what each permission does.
Week 2
Amazon Connect Hands-On Create a free Amazon Connect instance (free tier allows 90 minutes/month). Create a simple contact flow with Set Contact Attributes, Invoke Lambda, and Transfer to Queue blocks. Create a test queue and routing profile. Enable flow logs. Test with the built-in chat simulator. This is the most valuable hands-on exercise you can do.
Week 3
Lambda + CloudWatch Deploy a simple Lambda function that calls a Salesforce API (use any public API). Give it an IAM execution role. Trigger it manually and view the logs in CloudWatch. Then break it intentionally (wrong IAM permission) and practice reading the CloudWatch error. This exercise mirrors exactly what happens in SCC-AC when a Lambda fails.
Week 4
Read the Full SCC-AC Docs End-to-End With your new AWS fundamentals, re-read the entire SCC-AC documentation from Prerequisites through all Appendices. You'll understand it 3x better than the first read. Pay special attention to: Appendix B (all Lambda function descriptions), the IAM policy JSON in Prerequisites, and the 5 sample contact flows in Appendix B.
Week 5
End-to-End SCC-AC Installation in a Sandbox Execute the full SCC-AC installation in a Salesforce sandbox + new AWS account. Follow this reference document section by section. When something fails, practice debugging via CloudFormation events + CloudWatch logs. This will likely take 2-3 days and you'll hit at least 3 of the pitfalls listed above. That's expected and valuable.
Week 6+
Feature Configuration + Advanced Topics Configure each SCC-AC feature: chat attachments (S3 CORS), post-contact summaries (Contact Lens + IAM), unified routing (Cases), Amazon Q assistant. Study Appendix D (minimum runtime IAM permissions) and implement least-privilege security. Set up CloudWatch alarm actions. Plan for the 90-day IAM credential rotation cycle.

Final advice: The biggest risk in SCC-AC projects is the division of AWS and Salesforce responsibilities. Clearly define upfront: who owns the AWS account, who owns IAM, who owns contact flow development, and who owns CloudWatch monitoring. Blurred ownership causes deployment delays and production outages. The IAM credential rotation (every 90 days) requires coordination between both teams and is the most commonly missed operational task.
